### 2. 핵심 정리 (내 언어로)

- API 유형에는 무엇이 있을까?
  - web API: 오늘날의 대부분의 API
    - REST(아키텍처 스타일): 웹API 아키텍처의 원칙과 설계 방식을 정의한 스타일
      - 자원-URI: 인터넷 상에서 자원을 식별하는 고유한 주소(URL 등이 있음.)
        - URI는 정보의 자원을 표현 -> GET /menbers/1
        - 자원에 대한 행위는 HTTP Method로 표현 -> DELETE /members/1
      - 행위-HTTP METHOD: 클라이언트 서버 간의 상호작용을 정의하는 표준 방법.
        - POST: 리소스 생성
        - GET: 리소스 조회
        - PUT: 리소스 수정
        - DELETE: 리소스 삭제
    - GraphQL(설계 방식): 클라이언트가 API와 상호작용하는 방법을 지정하는 오픈소스 쿼리언어 및 서버 측 런터임.
    - WebSocket(프로토콜): 클라이언트와 서버간의 양방향 통신을 가능하게 함. / 실시간 커뮤니케이션에 이상적임.
    - SOAP(단순 객체 액세스 프로토콜): XML 메시지 형식을 사용 / HTTP 또는 SMTP로 소통
    - gRPX(구글 원격 프로시저 호풀 / 프레임워크): 구글에서 개발한 오픈소르 RPC(원격 프로세저 호출) 프레임워크
    - 왜 필요핡까?
      - 시스템간 연결 및 통합
      - 보안 강화
      - 새로운 수익 및 혁신 창출
      - 개발 생산성 향상
  - 데이터 API: 애플리케이션 & 데이터베이스 관리 시스템
  - 원격 API: 다양한 디바이스의 애플리케이션 상호작용 방식 정의

  - 상태 코드 Status Code: 클라이언트가 서버에 HTTP 요청을 보냈을 때 서버가 그 요청을 어떻게 처리했는지 숫자로 알려주는 응답코드
    - 1XX -> 정보
    - 2XX -> 성공 (200 OK: 요청이 정상적으로 처리됨 / 201 Create: 요청이 성공적으로 처리되어 리소스가 새로 생성됨)
    - 3XX -> 리다이렉션
    - 4XX -> 클라이언트 오류 (401 Unauthorized: 인증이 필요하거나 인증 실패 / 403 Forbidden: 서버가 요청을 이해했지만 권한 부족 등을 거부 / 404 Not Found: 요청한 레소스를 찾을 수 없음)
    - 5XX -> 서버 오류 (502 Bad Gateway: 게이트웨이/프록시 서버가 잘못된 응답을 받음)

- 데이터 제공 형식에는 어떤것이 있을까?
  - 데이터를 추출하기 위해 API를 통하여 JSON, XML과 같은 구조화된 형식의 데이터를 가져올 수 있다.
  - JSON(JavaScript Object Notation): JavaScript 객체 문법으로 구조화된 데이터를 표현하기 위한 문자 기반의 표준 포맷
    - HTML, Java, React -> 코드로 저장하고 실행하기 위한 포맷
    - JSON, XML -> 데이터 교환 하고 저장하기 위한 포맷
  - XML(EXtensible Markup Language): 데이터를 정의하는 규칙을 제공하는 마크업 언어
    - 마크업 언어
    - 데이터를 보여주지 않고, 데이터 전달과 저장만을 목적으로 함

  - API 명세서: API가 어떻게 이루어져 있는지 자세히 적어놓은 문서 == 백엔드-프론트엔드가 데이터를 주고받기 위해 미리 정해놓은 약속 문서
    - Endpoint: 어디로 요청을 보내야 하는지 (API PATH)
    - HTTP Method
    - Request Body 요청 데이터
    - Response 응답 데이터
    - Status Code 응답 상태 코드

  - API 테스트 도구
    - Postman: API를 로컬에서 테스트 할 수 있음.
